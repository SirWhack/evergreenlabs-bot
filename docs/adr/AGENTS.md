This scope documents the architectural decisions for the evergreenlabs-bot system, a cloud-native bot that transforms GitHub webhook events into an auto-updating personal website. The ADRs collectively describe a migration from a hybrid local/cloud architecture to a fully serverless system running on Cloudflare Workers, D1, and Workflows. The core responsibility is ingesting GitHub events (push, repository creation), running LLM-powered pipelines to draft site content, and publishing rendered `siteData.js` to a website repository via the GitHub Contents API.

The key architectural pattern is a **webhook-triggered Workflow with per-repo singleton debounce**, replacing the previous KV queue and Python polling approach. When a GitHub webhook arrives, a Worker verifies the signature, deduplicates by delivery-id using D1, and calls `WORKFLOW.create` with the repo name as the instance ID. This ensures only one Workflow instance runs per repository at a time, coalescing burst pushes within a ~30s window before drafting and publishing once. The end-to-end pipeline flows: webhook → Workflow → D1 cursor read → fetch commits → LLM judge + draft → D1 write draft → auto-accept → render siteData.js → Contents API publish.

The system exposes three key interfaces. An **MCP server** (Model Context Protocol) provides tools for agent-driven project management: `get_repo_context`, `list_items`, `get_site_status`, and sync triggers (`run_daily_sync`, `run_repo_workflow`). These tools are protected by a dedicated `MCP_TOKEN` secret, separate from the `TRIGGER_TOKEN` used for the manual HTTP endpoint at `POST /trigger/daily-sync`. The MCP server intentionally exposes no write access to site content (D1 `site_parts` or `drafts` tables), maintaining the bot's automated pipelines as the sole authority for site content. A cron trigger at 03:00 UTC runs the daily-sync workflow (project_sync → introduce backstop → roadmap_sync → publish).

Internally, the system uses **D1 as the single mutable state store** with tables for `site_parts`, `drafts`, `skipped_repos`, `cursors`, and `webhook_dedup`. Data is stored as JSON blobs per site part, rejecting relational schemas because the data shape is designed for the website renderer. The LLM provider is **OpenRouter** (chosen for speed and ~$1/mo cost), with the model selectable via environment variable. The MCP tool descriptions carry enough intent for agents to know when to use each tool, without exposing internal implementation details. The build plan follows a sequenced approach: scaffold, log_drafter vertical slice, daily-sync Workflow, port remaining pipelines, then cutover by flipping the GitHub App webhook URL and deleting the Python implementation.
